Characterizing the Software Process A Maturity Framework

改进(提升)++软件开发组织(software development organizations)++ 的表现十分重要。而这个改进过程一般有5个基本元素：

  • 1 了解开发过程的当前状态
  • 2 对目标过程有明确的愿景(a vision of the desired process)
  • 3 需要改进的优先级列表
  • 4 完成这些行动的计划
  • 5 执行计划的资源和承诺


本文通过为软件组织改进提供一个模型来解决前三个元素。 该模型的结构提供了五个成熟度级别，确定了每个级别所需的关键改进，并为实现建立了优先级顺序。这个模型已经在许多组织中进行了测试，发现它能够合理地表示实际软件开发组的状态和需求。

关键词：软件组织改进、改进模型

1. Introduction

如今软件成本以每年12%的速度增长，对软件具有更多功能的需求也日益增长。软件是目前美国国防部采购成本中一个重要且不断增加的部分，软件严重影响许多武器系统开发的进度和实用性。


本文描述了SEI项目的初试结果，该项目为美国国防部提供了一个描述软件开发组织能力的方法。 该项目提出的软件处理成熟度框架(the software process maturity framework) 可以被国防部或者软件开发组织来评估他们自己的能力，并确定最重要的改进区域

关键词：SEI项目，用来帮助软件开发组织评估自己的软件开发能力，并定位软件开发过程最重要的改进区域。

2. Software Development Processes

处理软件问题的一个重要的初始步骤是将整个开发任务视为一个可以控制、度量和改进的过程。 为此我们将++过程(process)++ 定义为一系列++任务(tasks)++，当这些任务被正确执行的时候就能产生想要的结果。显然，一个完整有效的过程必须考虑所有必须任务、使用的工具和方法以及相关人员的技能、培训和动机之间的相互关系。

“软件过程为一个为建造高质量软件所需完成的任务的框架，即形成软件产品的一系列步骤，包括中间产品、资源、角色及过程中采取的方法、工具等范畴。”—百度百科

这一段话的内在逻辑在：首先要把软件问题视为一个过程，而过程又可以被一系列任务定义。所以考虑软件过程重点就是要考虑这一系列任务(除了任务本身也包括执行任务的工具、执行任务的人啊之类的)

软件过程管理的基本原则是，如果开发过程处于统计控制之下，只有对过程进行改进才能产生始终如一比较好的结果。所谓++统计控制(statistical control)++ 指的是如果以大致相同的方式做同样的工作，就能得到大致相同的结果。

为了提高软件的能力，软件开发组织需要采取以下5个步骤：

  • 了解开发过程的当前状态；
  • 对期望的过程构建一个愿景(develop a vision of the desired process)
  • 按照优先级确定一个过程改进行动列表
  • 指定一个计划来完成这些行动
  • 提交资源来执行计划

本文就是通过提出一种将软件过程(软件过程的描述上文有)描述为5个成熟度级别之一 来解决上述问题。选用这种成熟度结构(maturity structure)有以下几个原因：

  • 选择成熟度级别是为了合理地表示实际软件开发组织改进的具体阶段
  • 每个成熟度级别代表软件改进过程的一个级别，该级别可以从先前的级别合理的实现。
  • 每个成熟度级别都应该建议中期的改进目标和进度措施
  • 一旦知道了组织在这个框架中的地位，应该立刻有一个改进优先级

这个成熟度结构 旨在和++评估方法++ 和 ++管理系统++一起使用。其中评估方法提供了一种方法来确定组织的特定成熟度状态，管理系统为实际实现改进组织所需的优先级操作建立了一个结构。

4. Process Maturity Levels

5个已经被被定义的成熟度级别为(下面会详细介绍)：

  • Initial - 只有在++统计控制(statistical control)++ 的过程，才可能有有序的改进过程
  • Repeatable - 通过对承诺、成本、进度和变更进行严格的项目管理，可以实现具有可重复统计控制级别的稳定过程。
  • Defined - 有必要对过程进行定义，来确保一致的实现，并且为更好的理解过程提供基础。在这一点上，先进的技术才可能被有效的引进。
  • Managed - 根据定义的过程，可以开始进行过程衡量(process measurement) 在一步开始出现最显著的质量改进。
  • Optimized - 通过被测量的过程，为过程的持续优化和改进提供了基础。

4.1. The Initial Process

初始过程通常是无序的。这里，组织通常没有正式的过程、成本估计和项目计划。工具也没有很好的和过程集成使用，对变更的控制也是松散的。由于问题常常被推迟，使之被遗忘，而不是被解决，所以软件安装和维护通常会出现问题。

组织用这种混乱的方式行事的一个关键原因是，他们没有获得足够的经验来理解这种行为的后果。在初始过程中的组织可以通过建立基本的项目控制来推进可重复的过程。最重要的是一下几点：

  • 项目管理。项目管理系统的根本作用是确保对承诺的有效控制。除了最简单的项目外，任何项目都必须制定计划，以确定最佳时间表和所需的预期资源。
  • 监督管理。一个适当的有纪律的软件开发组织必须有公司的监督。这包括在正式承诺之前审查和批准所有主要发展计划。按项目对整个设施的季度评审。缺乏这种审查会导致进程执行不平衡和不充分。
  • 产品保证。产品保证小组负责向管理层保证软件开发工作实际上是按照预期的方式完成的。
  • 控制变更。控制软件开发中的变更对于业务和财务控制以及技术稳定都是至关重要的。为了按照可以测的进度开发高质量的软件，必须在整个开发周期中以合理的稳定性建立和维护需求。

4.2. The Repeatable Process

可重复的过程相对于初始过程有一个重要的优势，它提供了一个合理的承诺控制度量。这对于初始过程中的performance来说是一个巨大的好处，以至于组织人员倾向于相信他们已经掌握了软件问题。他们没有意识到他们的力量来自于以前做类似工作的经验。因此，可重复过程中的组织在面临挑战时会遇到很多风险。一些改变的例子如下：

  • 每一个新技术都有风险和好处
  • 当可重复的过程组织必须开发一种新的产品时，它就进入了一个新的领域。会涉及的类似接口啊，集成问题之类的问题。
  • 重大的组织变更可能具有高度破坏性。

从可重复的过程推进到已定义的活动所需要关键行动如下：

  • 建立一个过程组。这是一个专门致力于改进软件开发过程的技术团队。
  • 建立软件开发过程体系结构，描述正确执行开发过程所需的技术和管理活动。体系结构是任务的结构化分解。每一个任务都有入口标准、功能描述、验证过程和出口标准。
  • 如果这样还没有到位，就引入一系列软件工程方法和技术。包括设计和代码检查、形式化设计方法、库控制系统和综合测试方法。

4.3. The Defined Process

通过已确定的过程，本组织已经取得了重大和持续进展的基础。尽管定义过程非常强大，但它只是定性的。也就是说，并没有数据表明正在发生什么，或者这个过程有多有效。关于软件度量的价值有很多争论。通过定义流程(defined process)，我们可以讲度量的重点放在特定的任务和项目上。

从已定义的过程过渡到以管理的过程，主要步骤如下：

  • 建立一个最小的基本过程测量集，以确定每一个过程步骤的质量和成本参数。
  • 利用资源建立流程数据库来管理和维护流程数据库。
  • 提供足够的资源来分析这些数据，并就数据的意义和使用向项目成员提供建议。
  • 评估每一种产品的相对质量，并告知管理层没有达到质量目标。

4.4. The Managed Process

从最初过程到托管过程的过渡中，软件组织通常会经历大量的质量改进。管理过程的最大潜在问题是收集数据的成本。有大量潜在有价值的软件开发数据，但是这些数据的收集和维护非常昂贵。

从管理过程到优化过程的两个基本需求是：

  • 为数据收集过程提供自动支持。
  • 将管理的重点从产品转移到过程。

4.5. The Optimized Process

从管理过程到优化过程的过程包括范式转换(paradigm shift)， 到目前为止，软件开发经理主要关注他们的产品工作，并通常只收集和分析与产品改进相关的数据。通过优化过程，数据可以用来调整过程本身。

有了优化过程，组织就有了方法来识别流程中最薄弱的元素并加以修正。人们不需要大量的纸张来描述发生了什么，因为简单的收益率曲线和统计图提供了清晰简洁的指标。

5. People in the Optimized Process

任何软件开发过程都依赖与实现它的人员的质量。优化过程在以下几个方面(对人员)提供帮助：

  • 1、它为管理部门提供了一种方法，可以让管理部门了解哪里需要帮助以及如何向人们提供他们所需的支持
  • 2、通过优化过程，专业人员可以通过简洁的量化术语进行沟通(意思就是极大减少了沟通成本)
  • 3、人们通常花费大量时间来纠正和弥补他人的错误。优化过程可以为专业人员了解他们的工作绩效和如何改进工作绩效提供一个框架。

优化过程为专业工作提供了一个有纪律的环境，但是很多程序员会对它对他们工作的影响感到紧张(理解理解，毕竟谁都不愿意在太disciplined的地方工作)，但是在高科技产业种，纪律是正常而自然的，不管是半导体设计师还是生物工程或者医学研究什么其他的领域，松散的工作环境都很难有创造力(都去玩手机了自然没有了)。

6. The Need for the Optimized Process

软件缺陷引发灾难的例子数不胜数，从火箭升空到导弹发射。如果不采取有序的步骤改进我们的软件过程，低质量代码的风险将变得难以承受。

软件质量不仅是管理问题，也是一个经济问题。你可以做更多的检查运行更多的测试。但是这样做既花费时间也花费金钱。只有经过优化的过程，产生的数据才能用来了解这种工作的成本和收益。因此，优化的过程为软件质量的显著提高以及同时提高生产力提供了基础。

7. Conclusion

这个软件开发成熟度模型 已经被证明合理的表现了软件开发组织改进的具体方法。它为++评估这些组织++和++确定立即改进的优先领域++提供了一个框架。它还帮助确定哪里的先进技术在改进软件开发过程中最有价值。
